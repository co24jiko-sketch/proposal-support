3つの「保存場所」の図解

Supabase・GitHub・Storage が何か、なぜ分けるのか

👤 ユーザー（担当者・部長など）
↓ ブラウザで操作
🌐 Webアプリ（技術提案書サポート）
Vercel で公開されている画面
↓ 中身によって保存先が違う
① Supabase（DB）
例えると → 業務台帳・Excel の表

何を入れる？

  • 案件名・発注者
  • 進捗（編集中・承認待ちなど）
  • 採点項目・適合チェック結果
  • 承認・差戻しの記録

なぜ使う？

  • 閉じても消えない
  • 案件ごとに変わるデータ向き
✓ いま使っている
② GitHub（コード）
例えると → 設計図・レシピ

何を入れる？

  • 画面の見た目
  • ボタンのルール
  • API の処理
  • デモ用サンプル

なぜ使う？

  • コードのバックアップ
  • Vercel がここから公開
✓ いま使っている
③ Storage（ファイル置き場）
例えると → Google ドライブ・書類棚

何を入れる？

  • 入札図書 PDF
  • 生成した Word
  • 出力した PDF

なぜ使う？

  • ファイルは DB よりこちらが向く
  • ダウンロードしやすい
これから使う予定

3つを並べて比べる

観点 Supabase GitHub Storage
入れるもの 文字・数字・状態
（表に書けるデータ）		 プログラム
（.tsx などのコード）		 PDF・Word など
（ファイル本体）
誰が変える？ ユーザーが
画面で操作すると変わる		 開発者が
コードを直すと変わる		 ユーザーが
アップロードすると増える
あなたのツール 案件データを保存 アプリの設計図を保存 まだ未実装

あなたのツールでは、こう動いている

1

GitHub に書いたプログラムを Vercel が読み取り、Web サイトとして公開する

2

ユーザーが「新規案件」を登録すると、Supabase の表（proposal_cases）に1行追加される

3

チェックリスト確定・適合チェック・承認も、すべて Supabase に保存される（F5 しても残る）

4

将来、Word をアップロードしたら Storage にファイルを置き、DB には「そのファイルの場所」だけ書く予定

DB ＝ 案件の中身 ／ GitHub ＝ アプリの作り方 ／ Storage ＝ ファイル
